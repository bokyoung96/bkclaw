# Acceptance Criteria

## A. Scope Acceptance
- [ ] Requested scope is written in one paragraph
- [ ] Out-of-scope items are explicitly listed
- [ ] Target files/modules are identified

## B. Harness Behavior Acceptance
- [ ] Harness follows defined execution stages
- [ ] Each stage has entry/exit condition
- [ ] Failure states are explicit
- [ ] Re-run behavior is defined
- [ ] Missing input behavior is defined

## C. Validation Acceptance
- [ ] Validation command(s) are documented
- [ ] Validation was actually run
- [ ] Output paths are captured
- [ ] Errors/warnings are recorded
- [ ] Result interpretation is attached

## D. Backtest / Metrics Acceptance
- [ ] Test period is specified
- [ ] Universe is specified
- [ ] Benchmark is specified or marked N/A
- [ ] Cost/slippage assumption is specified
- [ ] Rebalance cadence is specified
- [ ] Core metrics are present
- [ ] Metric source file/path is recorded
- [ ] Chart/media artifact exists when requested

## E. Discord Delivery Acceptance
- [ ] Start/progress/final reporting happened for long task
- [ ] Final summary message exists
- [ ] Media uploaded if user asked for visible chart/file
- [ ] Delivery proof exists (message id / CLI success output)
- [ ] “Sent” is not claimed without proof

## F. Completion Acceptance
- [ ] Completion proof document is filled
- [ ] Remaining uncertainty is listed
- [ ] Next action is stated
- [ ] No vague “done” without evidence
